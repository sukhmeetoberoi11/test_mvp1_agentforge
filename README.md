# Customer Support Triage System

> Generated by [AgentForge](https://agentforge.io)

## Overview

The Customer Support Triage System is a supervisor-worker multi-agent platform built on LangGraph, where a central Coordinator Agent classifies inbound support tickets by intent and urgency, then routes them via Send() to specialist Worker Agents (Billing, Technical, General). All agent state is externalized to Redis for durable, resumable multi-turn conversations, while Qdrant provides semantic retrieval over knowledge base articles, policy documents, and historical tickets. Every draft response produced by a worker passes through a Compliance Reviewer Agent that enforces PII cleanliness and policy adherence before delivery, with a reject-and-retry loop (max 2 cycles) and automatic human escalation on breach.

Four Claude-3-5-Sonnet-backed agent nodes share a typed SupportTicketState schema covering ticket identity, classification outputs, conversation history, tool call logs, draft responses, and compliance metadata. Tool access is RBAC-scoped at the wrapper layer: billing_system writes are restricted to the Billing Agent, account_database writes to Billing and Technical Agents, and ticket_management_system writes to the Response Emitter node only. PII is detected and tokenized at two checkpoints — ingestion and pre-emit — with original values stored in a separate encrypted vault never written to Redis or Qdrant.

At medium load (500–2,000 tickets/hour), the architecture scales horizontally via stateless LangGraph worker processes behind a load balancer, all sharing a Redis Cluster and a distributed Qdrant instance. Inbound tickets are enqueued via a message queue (Redis Streams or SQS) for backpressure handling. Async coroutine execution within each worker process enables concurrent ticket processing, and Anthropic prompt caching on all four agent system prompts reduces token cost and latency. The target end-to-end P95 latency is approximately 4.5 seconds across classification, worker execution, compliance review, and tool calls.


## Deployment

This repository includes two GitHub Actions workflows — **Azure Container Apps** (default)
and GCP Cloud Run (available but inactive).

### One-time Azure setup (automated on connect)

When you call `POST /github/connect` with AgentForge's Azure admin credentials configured,
AgentForge automatically configures OIDC and writes AZURE_* secrets to the GitHub
**Environment** named in `AZURE_GITHUB_ENVIRONMENT` (default: `production`).

The workflow (`.github/workflows/deploy-azure.yml`) uses `environment: production` and
is triggered only by `POST /deployments/{agent_id}` (or manual **Run workflow** in GitHub).

### GCP Cloud Run (manual only)

A `.github/workflows/deploy-gcp.yml` is present for future use. It does **not** run on
push — only via **Actions → Deploy to Cloud Run → Run workflow**. To use it, add three
GitHub Secrets and set `DEFAULT_DEPLOYMENT_PROVIDER=gcp` in AgentForge:
- `GCP_WORKLOAD_IDENTITY_PROVIDER`, `GCP_SERVICE_ACCOUNT`, `GCP_REGION`
